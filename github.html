Gérez votre code avec Git et GitHub

Lors de la deuxième année à l’EPHEC, pour le projet en Java, j’ai utilisé l’outil github pour abriter mon code. J’ai reçu une petite explication en cours mais je trouvais que cela n’était pas suffisant pour l’utiliser à sa toute puissance. J’ai donc décidé de voir par moi-même le fonctionnement avec de la pratique avant le lancement du projet. L’utilisation de github était vraiment nouvelle pour moi et j’avais besoin de pratique pour bien le maitriser. J’ai donc trouvé la formation proposée sur openclassroom et je m’y suis lancé.

Différents concepts sont liés à github. Chacun indispensable pour travailler en groupe sur un même projet. Le système de versionning, branches, commit, pull, etc.

Le concept le plus important est le versionning. Il permet de garder des traces de ce qui est fait. Ces traces s’appellent des commits. Chaque commit présente les modifications réalisées. Par exemple l’ajout/suppression d’un fichier, du texte ajouté ou autre. Grâce à ces commits, un historique est créé et il est possible de naviguer parmi ces commits. Chacun est accompagné d’un petit message permettant de connaitre son utilité. Au final les outils de versionning sont devenus indispensables à la bonne réalisation d’un projet en groupe. Surtout dans le monde de l’informatique.

Jeff Atwood dit même que « si un code n’est pas enregistré dans un logiciel de gestion de version alors il n’existe pas », cela prouve bien son intérêt.

J’en ai parlé plus haut mais les commits gardent des traces et en plus permettent de savoir qui a fait ces modifications. Le but est de faire des modifications (pas trop peu, ni beaucoup trop). Ensuite l’utilisateur fait son commit et le code commence à être versionné. Lors d’un commit, il est possible de voir les modifications fait sur chaque fichier et ainsi de s’assurer que les modifications sont bonnes. Après plusieurs commit il est possible de revenir en arrière. Si un problème survient ou alors que les modifications ne sont pas intéressantes, je choisis de revenir à un commit fait précédemment. Cette fonctionnalité permet d’avancer sans crainte de casser son code.

Déjà avec ces concepts le projet peux avancer d’un bon pas mais github va plus loin que ça. Github est un système partagé donc il est possible de garder un backup de son code ainsi qu’un endroit de partage pour sauvegarder son code. Avec ce concept, deux autres apparaissent, le pull et le push.

Tout d’abord le pull, celui-ci permet de récupérer le code déposé sur github. En faisant un pull je télécharge le code sur ma machine et je peux librement faire mes changements.

Ensuite le push, dès que j’ai finis mes modifications, je commit pour signaler mes changements et push tout cela pour que les autres puissent profiter de mes changements. Ce système permet d’éviter les conflits entre le travail de chaque personne participant au projet.

Dernier concept intéressant ce sont les branches. Pour s’assurer d’un projet en production toujours opérationnel, les personnes vont travailler sur leur propre branche et mettre tous leurs changements sur une branche dev. Tous les tests vont se faire sur celle-ci. Par la suite, les changements sont ajoutés au prod.
